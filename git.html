Git

Controle de versão

VCS Sistema de controle de versão, como isso podemos, registrar alterações em arquivos com a finalidade de não perder-los ou reverter para um estado anterior,também é possível ver oque foi modificado e muito mais.

Tipos de Controle de Versão

  • Sistemas Locais

    Copiar os arquivos para outro diretório. ( Sem um sistema de versionamento. ) Criando arquivos manualmente, um sistema manual.

  • Sistemas Centralizados

    Exemplos: CVS, Subversion e Perforce.

    Sistemas Centrilazados será centralizado em um único servidor, podendo vários clientes usarem arquivops desse mesmo servidor. Porém há vantagens e desvantages:

    • Vantagens

      -Controle sobre atividades dos colaboradores no projeto.
      -Os administradores têm controle refinado sobre quem pode fazer o que;
      -Mais fácil de administrar um CVCS do que lidar com bancos de dados locais em cada cliente.

    • Desvantagens

      -Se esse servidor der problemas, ninguém pode colaborar ou salvar alterações de versão.
      -Se o disco rigído do banco de dados for corrompido, perderá tudo. -Sistemas VCS locais sofrem com esse mesmo problema.

  • Sistema Distribuídos

    Exemplo: Git, Mercurial, Bazaar ou Darcs.

    -Ele duplica ( clona ), localmente o repositório completo.
    -Se qualquer servidor morrer, os repositórios estarão com os colaboradores, cada clone é de fato um backup completo de todos os dados.
    -Clientes usam o estado mais recente dos arquivos.

O que é o Git.

O git é um VCS, open soucer.

Com ele é possível registrar "Pontos" no código/arquivo para voltarmos se for necessário, seja por um erro ou necessidade adiversas, contém histórico de alterações no código e entre outros como controlar fluxo de novas funcionalidades.

É possível criarmos branch, ramificações dos "Pontos", para não afetar o ponto funcional e assim podendo outros Devs trabalhar juntos.

Configurando Git - Primeira vez

Comandos git assim que instalar.
Adicionar nosso nome: git config --global user.name "Seu Nome Completo aqui". O nome entre aspas.
Adicionar nosso email: git config --global user.email emailteste@outlook.com. O email não precisa das aspas.

Entendo os comandos

O comando git config nos permite ver e atribuir variáveis de configuração como nome e email.
Estas variáveis podem ser armazenadas em três lugares diferentes:

  1. /etc/gitconfig: Válido para toso os usuários no sistema e todos os seus repositórios. Se você passar a opção --system para git config, ele vê e escreve neste arquivo.

  2. ~/.gitconfig ou ~/.config/git/config Somente para o seu usuário. Você pode fazer o Git ler e escrever neste arquivo passando a opção --global

  3. config no diretório Git (ou seja, .git/config ) de qualquer repositório que você esteja usando: específico para este repositório.

Cada nível sobrescreve os valores do nível anterior, ou seja, valores em .git/config prevalece sobre /etc/gitconfig.

No Windows, o arquivo .gitconfig estará no diretório $HOME que é C:\Users\$USER

GIT HELP.

Comando git help, podemos ver muita coisa sobre o git, para abrender e entender como funciona o git e seus comandos.

Git log

Todo log tem um commit seguido pelo nome e onde estamos qual a branch ( ramifiação ).

Comandos legais de ter:

git log - Trás todo o log.
git log --oneline - Trás informações importantes do logo apenas.
git log -n x - Trás o log do valor apontado no "x".
git log --since="2020-10-01 - Trás o log após o valor informado.
git log --until="2020-10-01 - Trás o log anterior ao valor informado
git log --author=Nome - Trás o log do author informado.
git log --grep"init" - Trás tudo que ele encontrará com o nome informado, no commit

Commit

Para adicionarmos um arquivo ao git, temos que usar o comando: git add . após isso git commit -m "initial commit" e pronto foi feito um commit.

Também sendo possível adicionarmos arquivos de uma única vez, determinando o tipo de sua extensão, comando: git add *.html.
Ou seja utilizando o "*" e após ele o tipo de arquivo, ele irá puxar todos com o mesmo tipo que foi determinado e irá para stage tree.

Hash Values (SHA-1)

Serve para garantir a integridade do commit, ele é gerado para garantir que não sejam repetidos o mesmo contém alguns meta dados, são eles:

  • parent
  • author
  • message:
Assim nunca terá as mesmas informações.

Head

Head podemos dizer que ele é um ponteiro, apontando aonde estamos como ( Master ).
Podemos pensar que o Head aponta para o ponto que estamos, por exemplo realizando um commit logo o Head apontará a este commit.

Git diff

É um comando que mostra as alterações nos arquivos.
Ele só mostrará os arquivos restreados ou seja, os arquivos que foram adicionados ao repositório.

Git remove

Possível excluirmos um arquivo do git, utilizando o comando: git rm "file" o mesmo irá adicionar ao stage area e pronto para comitarmos a exclusão do arquivo.

Renomear arquivos / Mover

Utilizando o comando git mv "nomedoarquivo" "novonomedoarquivo" fazemos a alteração do nome, desta forma é necessário apenas commitar a mudança.

O mesmo comando serve para mover o algum arquivo, mudando apenas o paramêtro, sendo: git mv "nomedoarquivo" "novaPasta/nomedoarquivo", adicionamos o nome do arquivo e o novo diretório do mesmo.

Desfazer Modificações

Utilizando o comando git restore "nomedoarquivo", funciona antes do stage area.

Comandossss em geral

Restaurar aquivo do stage area, podemos usar o comando: git restore --staged "nomedoarquivo", também é possível restaurarmos vários arquivos de uma única vez, usando o "." no final, ou seja.
git restore --staged .

Restaurar o último commit

Usando o comando: git commit --amend -m "nomedamensagem" com esta comando é possível mudarmos o último commit feito, ou seja mudou a mensagem do último commit e seu HASH também passar a ser um novo.

Restaurar aquivos

Para recuperarmos um arquivo do git precisamos usar o código: git checkout -- "nomedoCommit" "nomedoArquivo", sendo o nome do Commit o código que é informado e o nome do arquivo em si.
Após isto podemos só retirar o arquivo do stage area para não ser comitado se for o objetivo, usando o comando: git restore --staged "nomedoArquivo".

Excluir arquivos não rastreados

É possível excluirmos arquivos não rastreados com o código: git-clean -n, este irá mostar oque será removido e para exlcuirmos de fato o código: git-clean -f. Tomar cuidado com o comando, pois o mesmo será deletado e impossível de ser recuperado.

Revertendo um Commit

É possível trazermos arquivos de um commit utilizando o código git revert HEAD~5, one o "HEAD~5" seria o 5 commit anterior ao último.
Também é possível trazermos os arquivos de volta utilizando o Hash do commit, exemplo: git revert "Hash do commit", assim já irá trazer os arquivos.
O próprio git fará um novo commit.

Apredendo usar o git em um projeto real.

Comandos que podem nos ajudar.

Após modificarmos um arquivo dentro de um repositório já existente, já rastreável, é possível utilizarmos o comando git commit -am "texto" ou seja, ele irá jogar o arquivo para stage area com o "-a" e com o "m" fará o commit, porém é possível juntarmos por isto ( -am "texto" )

Ver modificação de um commit

É possível vermos a modificação de um commit utilizando o código: git show "hash".
Também é possível vermos as modificações de um diretório com o código: git show "hash" -- "diretório ( exemplo: src/views/*)".
